## Architecture decisions

### Backend agent model
The existing architecture: agent MD files in `src/agents/prompts/` are parsed via `gray-matter`. Frontmatter = config (inputs, tone, est). Body = system prompt sent to OpenRouter. The `AgentsService` maps each agent ID to a Jira data route (`sprint`, `tickets`, `epic`). The controller fetches Jira data, builds a user message (Jira JSON + form values), then streams the LLM response as SSE.

`bug-report` reuses the existing `epic` route ‚Äî it fetches `getEpicWithChildren(epicKey)` which returns the epic + all child issues. The LLM sorts through bug types in the data. Toggle states arrive as form values (`addon_bug_report=true`, `addon_bug_analyzer=false`, etc.) and the system prompt conditions generation on those values.

### Sprint ID vs Sprint Name
Jira's Agile API returns sprint objects with integer IDs. JQL supports `sprint = {id}` (no quotes) for ID-based queries, which is more reliable than name matching. The `jira-sprint` input stores the sprint ID as string. Backend reads `formValues.sprint_id`, converts to `Number`, and uses it in JQL if present.

### Dynamic input types (frontend)
Three new input types added to the RunPanel renderer:
- `jira-project` ‚Üí `JiraProjectSelect` reads from `localStorage['JIRAPILOT_WORKSPACES']`
- `jira-sprint` ‚Üí `JiraSprintSelect` fetches `POST /jira/sprints` when `dependsOn` field changes
- `toggle` ‚Üí `ToggleInput` (pill-style switch); rendered in a separate "Add-ons" section below regular inputs

`dependsOn` is a frontmatter field on the `jira-sprint` input that names the ID of the project input whose value to pass to the sprints fetch.

## Key constraints / notes

- `JIRA_BASE_URL` env var drives the base URL for all Jira HTTP calls (set in BE `.env`)
- The Agile API (`/rest/agile/1.0/...`) shares the same base URL as the REST API ‚Äî no separate config needed
- `JiraTokenGuard` (global) requires `X-Jira-Token` header on all routes; the new endpoints comply
- `bug-report` blocker predictor section explicitly excludes UI styling, spacing, and translation bugs from blocker classification ‚Äî this is baked into the system prompt
- `JIRAPILOT_WORKSPACES` is populated lazily on first successful connection test; until then `jira-project` inputs show a "test your Jira connection" placeholder
- Toggle default values initialise via the existing `seed[f.id] = f.default || ''` loop in `RunPanel`; `"true"` / `"false"` strings are used throughout, never booleans, because all form values are strings
